Repo bootstrap: pnpm workspace, Anchor program, connection utility, and a full local/CI dev loop

This branch creates the entire game-token-wallet-v2 repo from an empty shell: a pnpm workspace with an Anchor program, a Codama-generated TypeScript client, a Next.js frontend, and two real (not stubbed) e2e test apps, all wired together by a Docker Compose local-dev loop and a CircleCI pipeline. The functional payload is deliberately trivial (one noop instruction) — the point of the ticket is proving every layer of plumbing (env config to signer to RPC to on-chain program to UI to CI) actually works end-to-end, since every later ticket builds directly on top of this scaffold.

Why

This is ticket 001 of the project: before any real game/token logic can be written, the team needs a repo that installs, builds, tests, and deploys reproducibly — and a single, disciplined place where the server-side Solana connection and system-admin signer are constructed, since the whole product is custodial (users never touch a wallet or seed phrase; the server signs on their behalf). Getting that discipline right on day one matters more than the feature itself: every later ticket will import from this connection utility rather than each inventing its own RPC/signer construction.

The noop instruction is intentionally the simplest possible on-chain action (no accounts, no args, no logic) so this ticket can prove the full chain — Server Action calls the utility, the utility loads env and builds a signer, the signer submits through the Anchor-generated client, the transaction lands on a real (Surfpool) validator, and the resulting signature renders in the browser — without any product logic getting in the way of validating the scaffold itself. The design doc and plan record that most structural decisions (repo tree, tech stack, account model) were already settled in prior sessions; this ticket resolves the remaining implementation-level choices (env var shapes, exact tool versions, verification approach) needed to actually build it.

Root workspace scaffold and tooling

The repo becomes a pnpm workspace (pnpm-workspace.yaml: apps/*, packages/*) with a root package.json exposing lint/typecheck/test/build/codegen scripts that fan out via pnpm -r --if-present run <script>. A shared tsconfig.base.json, a flat ESLint config (eslint.config.mjs), and Prettier config give every workspace package the same baseline instead of each reinventing it.

The --if-present fan-out has a sharp edge that review caught late (round 3): a package that simply lacks a lint/typecheck script is silently skipped rather than failing, so for most of the loop 3 of the 4 TypeScript packages had no real type-checking in CI even though the root script reported success. That was closed by giving every package its own lint/typecheck script (and a tsconfig.json for apps/e2e, which had none at all), turning a silent gap into an enforced one.

Anchor program skeleton (game_token_wallet + noop)

apps/on-chain-program is an Anchor 1.x program (chosen over the older 0.30.x line specifically because Anchor 1.0 made Surfpool/LiteSVM the default test backend, matching the already-chosen local-dev strategy) with a single crate game_token_wallet exposing one instruction, noop, that takes no accounts and no arguments and does nothing (Ok(())). The plan's own placeholder declare_id! was malformed (41 characters instead of a valid 32-byte pubkey) and had to be fixed just to get anchor build running before anchor keys sync overwrote it with the real generated address anyway.

Because noop has zero logic, the ticket deliberately does not write a contrived Rust unit test for it (a documented deviation from the original spec's §8) — instead cargo test passing with its one auto-generated test_id test (from the declare_id! macro) is treated as this task's gate, and the instruction's real coverage comes later from the on-chain integration test (Task 4) and the Playwright e2e test (Task 8), both of which invoke noop against a live validator.

Codama codegen (packages/on-chain-client)

Rather than hand-writing a TypeScript client for the Anchor program, packages/on-chain-client runs Codama (@codama/nodes-from-anchor + @codama/renderers-js) against the built IDL (apps/on-chain-program/target/idl/game_token_wallet.json) to generate getNoopInstruction and a GAME_TOKEN_WALLET_PROGRAM_ADDRESS constant. The generated output under src/generated/ is committed to git rather than gitignored — a deliberate call to sidestep Docker build-ordering problems between the program-deploy step and the frontend image build (the frontend image can't wait on a sibling container's codegen step during its own build).

A real signature mismatch surfaced during implementation: the plan assumed getNoopInstruction(input, config) (two arguments), but the actual generated function takes a single config object, getNoopInstruction({ programAddress }). This was caught in Task 3 and had to be threaded through as a fix in both Task 4's on-chain test and Task 7's Server Action, since both call sites had copied the plan's stale example.

The unit test for this package (src/index.test.ts, checking that the export is a function and the program-address constant is a non-empty string) was itself dead code for most of the loop — round 1 (F1) found neither CircleCI nor the Justfile ever invoked pnpm --filter on-chain-client run test, so a real, committed test suite simply never ran. It was wired into CI's web-unit-tests job and the Justfile's test recipe.

Connection/signer utility (apps/frontend/src/server/) — the ticket's core deliverable

env.ts (loadSolanaEnv) reads four required env vars (SOLANA_CLUSTER, SOLANA_RPC_URL, PROGRAM_ID, SYSTEM_ADMIN_SECRET_KEY) with no silent fallback — any missing or unrecognized value throws immediately with a specific message naming the offending var. connection.ts (getSolanaContext) builds on top of that: it decodes the base58 admin secret key, validates it decodes to exactly 64 bytes (rejecting silently-wrong keys rather than constructing a garbage signer), and memoizes the resulting { rpc, rpcSubscriptions, adminSigner, programAddress } context in a module-level promise so it's built exactly once per process and reused thereafter. This file is the only place in the codebase permitted to construct an RPC client or signer — every other server module (Server Actions, future instruction wrappers) must import from here.

Both files were built TDD-first: failing tests were written before the implementation for both the happy path and edge cases (missing var, malformed secret key, invalid cluster name, and — for the connection utility — that repeated calls return the identical object reference, proving the singleton). One bug surfaced in the plan's own fixtures during this task: the literal 41-character PROGRAM_ID placeholder used in the tests didn't decode to a valid 32-byte pubkey — connection.test.ts's real address-decoding assertion caught it where env.test.ts's non-decoding toEqual comparison didn't, and both test files were corrected. The implementation code itself matched the plan verbatim.

A real library gap also had to be worked around here rather than papered over: @solana/kit@5.5.1's signTransactionMessageWithSigners returns a non-generic TransactionWithLifetime, which isn't assignable to what sendAndConfirmTransactionFactory expects. Rather than an as cast (which would lie to the type system), the code calls assertIsTransactionWithBlockhashLifetime — a documented runtime type-guard from the library itself — to narrow the type by actually checking it at runtime.

Self-check

What four env vars does loadSolanaEnv require, and what happens if SOLANA_CLUSTER is set to "testnet"?
SOLANA_CLUSTER, SOLANA_RPC_URL, PROGRAM_ID, SYSTEM_ADMIN_SECRET_KEY — all required with no defaults; SOLANA_CLUSTER=testnet throws because only localnet | devnet | mainnet-beta are recognized.
env.ts's CLUSTERS constant is the enum; loadSolanaEnv throws Invalid SOLANA_CLUSTER "..." for any other value, matching env.test.ts's corresponding test.
Why does getSolanaContext() memoize a Promise<SolanaContext> rather than a resolved SolanaContext?
Because building the context requires an async step (createKeyPairSignerFromBytes), so the singleton has to cache the in-flight promise — caching only after resolution would let concurrent early callers race and build the context twice.
connection.ts sets contextPromise = createContext() synchronously on first call, before awaiting anything, so every caller (even concurrent ones) gets the same promise instance.
What real problem does decoding-and-length-checking the admin secret key protect against, and which test file's assertion actually exercises pubkey decoding versus which one doesn't?
It protects against silently constructing a garbage signer from a malformed key. connection.test.ts actually decodes the address (address(env.programId)), which is what caught the plan's invalid 41-character placeholder pubkey; env.test.ts's toEqual on the raw string didn't.
decisions.md (task-6) documents this exact discrepancy: the implementation was correct as planned, but the plan's own test fixture data had a bug only one of the two test files was capable of catching.
Why does sendNoopTransaction (and the on-chain-program-e2e test) call assertIsTransactionWithBlockhashLifetime instead of casting the signed transaction's type?
signTransactionMessageWithSigners returns a non-generic TransactionWithLifetime type that doesn't statically satisfy sendAndConfirmTransactionFactory's required blockhash-lifetime type, and a cast would silently hide a real mismatch; the assertion performs an actual runtime check instead.
This is a genuine @solana/kit@5.5.1 type gap, independently reproduced against already-committed code before being accepted as a library issue rather than implementer error (task-7 decisions).

noop Server Action, proof-of-plumbing page, and the two e2e skeleton apps

apps/frontend/src/server/actions/noop.ts is a "use server" Server Action (sendNoopTransaction) that pulls the memoized context from getSolanaContext(), builds a versioned transaction with the generated getNoopInstruction, signs it with the admin signer, sends and confirms it, and returns the resulting signature. apps/frontend/src/app/page.tsx is a client component with one button that calls this action and displays either the returned signature or the error message — this is deliberately the whole UI for this ticket, since the goal is proving the wiring, not building product UI.

Two separate 'leaf' test apps exercise noop for real rather than being placeholder CI stanzas: apps/on-chain-program-e2e is a vitest suite invoked by anchor test (via Anchor.toml's [scripts] test) that airdrops SOL, builds a transaction with the generated client, and confirms it against the validator Anchor itself spins up. apps/e2e is a Playwright test that drives the actual running frontend through the browser — navigates to /, clicks the button, and asserts a valid-looking base58 signature renders. Both give this trivial instruction genuine, non-vacuous coverage across two different layers (program-level and full-stack-through-the-browser).

Local Docker dev loop and the program-deploy-keypair reproducibility bug

docker-compose.yml is the single local entry point (just dev-up): surfpool (a pinned-by-digest validator image) becomes healthy, program-deploy (a one-shot Anchor build+deploy+codegen container) runs to completion, then frontend boots against the deployed program. Getting the program-deploy image working at all required abandoning the plan's assumptions: the original debian:bookworm-slim base couldn't work on Apple Silicon because neither Solana CLI nor Anchor CLI publish linux/arm64 binaries, so the image is forced to platform: linux/amd64; the plan's one-line mega-installer script panicked with no TTY in Docker, so it was replaced with explicit rustup/Solana-installer/cargo install avm steps; and the base image had to move from bookworm to trixie because anchor-cli 1.1.2's prebuilt binary needs a newer glibc than bookworm ships. Named Docker volumes were also added to isolate each workspace package's node_modules, because a host (arm64) and container (amd64) both writing into the same bind-mounted node_modules corrupted native binaries badly enough to crash a live dev server mid-request.

The most significant bug in the whole diff surfaced only during live re-verification, after the rest of the stack looked done: the program's deployed identity depended on a gitignored, host-mutable file, target/deploy/game_token_wallet-keypair.json. Whenever that file went missing (a fresh clone, or another anchor build run elsewhere on the same host — which had actually happened across the many agent runs sharing this host's checkout), Anchor silently minted a brand-new random keypair and the deployed program's address quietly drifted out of sync with the declare_id!/Anchor.toml/PROGRAM_ID values already committed to source. The fix commits a fixed, throwaway, non-secret keypair (docker/local/fixtures/game_token_wallet-keypair.dev.json, address FHRNx4KK4WzMxXx7X6sK84RvKTKuDVtTGduW3eH9QN9t), which the Dockerfile/compose command now seeds into target/deploy/ before every anchor build runs — the same "commit a throwaway dev secret" pattern already used for SYSTEM_ADMIN_SECRET_KEY. It was proven reproducible by deleting the local keypair and rebuilding fresh (same address every time) and independently re-confirmed by a separate verifier agent submitting a real on-chain transaction.

A second, smaller funding bug was also found and fixed here: the plan only airdropped SOL to the container's own ephemeral CLI keypair, leaving the actual SYSTEM_ADMIN_SECRET_KEY fee-payer wallet unfunded — every noop transaction from the frontend would fail with "no record of a prior credit." The fix adds a second solana airdrop targeting that specific wallet address.

CI-parity e2e stack (docker-compose.e2e.yml)

This is a second, prod-parity compose stack reusing docker/local/Dockerfile.anchor as-is but building the frontend from a real multi-stage production Dockerfile (docker/deployment/Dockerfile) instead of the local dev image, and adding a fourth service, e2e, that runs the Playwright suite in a container against that prod-built frontend. This is exactly the command CircleCI's e2e job runs.

Getting this stack to actually finish required working around a real Docker Compose limitation, not an application bug: --abort-on-container-exit (needed so the overall docker compose up command exits with the test's real pass/fail code) tears down the entire stack the instant any container exits — including program-deploy, which is supposed to exit 0 on success. The original one-shot design reproducibly tore down surfpool mid-flight before frontend/e2e could even start, failing with "getaddrinfo ENOTFOUND surfpool." The fix keeps program-deploy alive forever after finishing its real work (tail -f /dev/null) and signals completion via a marker file plus a healthcheck, so frontend's depends_on: condition: service_healthy can gate on it without anything ever exiting early. A profiles:-based alternative was tried first and rejected because docker compose config didn't accept it. A production Dockerfile bug was also found on the first attempt: it omitted COPY tsconfig.base.json, breaking Turbopack's tsconfig resolution during the prod build.

Self-check

Why can't program-deploy simply exit 0 after finishing its build/deploy/codegen work in this compose file, even though that's exactly what it does in the plain docker-compose.yml?
Because this stack is run with --abort-on-container-exit --exit-code-from e2e, and Compose's --abort-on-container-exit tears down every other container the moment ANY explicit container exits — even a one-shot dependency that exited 0 on purpose — which reproducibly killed surfpool before frontend/e2e could run.
docker-compose.e2e.yml's inline comment cites this as a documented Compose limitation (linked GitHub issues docker/compose#10233, #13223), confirmed empirically in Task 10, not something fixable by changing program-deploy's own script.
How does frontend know program-deploy has actually finished, given that program-deploy never exits in this file?
program-deploy touches a marker file (/tmp/deploy-complete) after its real work finishes, and its healthcheck polls for that file's existence; frontend's depends_on: program-deploy: condition: service_healthy gates on that healthcheck instead of service_completed_successfully.
Visible directly in docker-compose.e2e.yml: command ends with touch /tmp/deploy-complete && tail -f /dev/null, and the healthcheck is test -f /tmp/deploy-complete.
What real bug did the first attempt at docker/deployment/Dockerfile (the production frontend image) have?
It omitted COPY tsconfig.base.json, which broke the Turbopack build with a tsconfig resolution error, since the frontend's own tsconfig extends the root one.
Recorded as Attempt 1's failure in task-10's "Challenges faced" in decisions.md.

CircleCI pipeline

.circleci/config.yml defines six real jobs on every PR (no fast-PR/slow-main split, per an earlier project decision): lint, typecheck, web-unit-tests (frontend + on-chain-client vitest suites), cargo-test, on-chain-program-e2e (real anchor test against a live validator), and e2e (the full docker-compose.e2e.yml stack). Toolchain installation is factored into two shared CircleCI commands: blocks — install_rust (bare rustup, used by cargo-test) and install_anchor_toolchain (rustup + Solana CLI + avm-built-from-source + pinned anchor-cli 1.1.2, used by on-chain-program-e2e) — split apart because cargo-test was confirmed to need no Solana/Anchor CLI dependency at all (verified by running cargo test with solana/anchor/avm stripped from PATH).

A real, if narrow, correctness gap review caught: cargo-test's job initially installed the full toolchain for a job that never touches it (round 1, F4) — trimmed to the minimal install_rust. A follow-on defect from that very fix then had to be caught in round 2 (F1): a comment justifying tolerating cimg/base:current's floating tag via "anchor-cli's GLIBC_2.39+ requirement" was left in place even after the fix removed anchor-cli from that job entirely, so the comment's own stated rationale had become false — it was rewritten to match the corrected job.

Known risks & deferred items

  • .gitignore's !apps/on-chain-program/target/idl/ negation is a no-op — git cannot re-include a path under an already-excluded parent directory; harmless today, inherited from the plan.
  • docker-compose.yml and docker-compose.e2e.yml duplicate the surfpool service config and dev-fixture literals (PROGRAM_ID, SYSTEM_ADMIN_SECRET_KEY) verbatim with no shared base — future edits to one could silently drift from the other.
  • The keypair-seed + build + deploy + codegen pipeline is duplicated between Dockerfile.anchor's CMD and docker-compose.e2e.yml's command override.
  • No CircleCI caching (pnpm store, cargo registry, toolchain installs) — cost/time only, not correctness.
  • Repeated pnpm-setup boilerplate copy-pasted across 4 CircleCI jobs instead of using the file's own established shared command: pattern.
  • web-unit-tests CI job serializes two independent test suites (frontend, on-chain-client) in one job instead of splitting into parallel jobs, inconsistent with how cargo-test/on-chain-program-e2e are split.
  • apps/on-chain-program-e2e's test hardcodes RPC/WS URL literals (8899/8900) duplicating the convention connection.ts's deriveWsUrl() already centralizes.
