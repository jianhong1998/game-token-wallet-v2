## task-9-local-dev-loop-docker-compose

### Key decisions
- Forced `platform: linux/amd64` on `program-deploy`/`frontend` — confirmed via GitHub Releases API that neither Agave nor Anchor v1.1.2 publish linux/arm64 binaries.
- Switched base image to `node:24.18.0-trixie` (not `-bookworm`) — anchor-cli 1.1.2's prebuilt binary needs GLIBC_2.39+, which bookworm's 2.36 doesn't meet.
- Dropped the plan's mega-installer script entirely (reproducibly panics with no TTY in Docker, and separately never sources NVM into subsequent RUN layers) in favor of explicit rustup/official Solana installer/`cargo install avm` steps.
- Used named Docker volumes (5) to isolate `node_modules` per workspace package from the bind-mounted repo — cross-platform (host arm64 / container amd64) `pnpm install` runs were corrupting each other's native binaries, up to crashing a live Next.js dev server mid-request.
- Set `CI=true` where needed — pnpm refuses a non-interactive stale-`node_modules` wipe without a TTY otherwise.
- Re-synced the program's declared address via `anchor keys sync` rather than forcing the git-committed address, since that keypair's private key wasn't recoverable in this session; propagated the corrected address everywhere.
- Added a second airdrop funding the frontend's `SYSTEM_ADMIN_SECRET_KEY` wallet — the plan only funded the container's own ephemeral CLI keypair, leaving the actual transaction fee-payer unfunded.
- Fixed a WS pub-sub URL bug (reusing the RPC port instead of Solana's `rpc_port + 1` convention) — later generalized into `deriveWsUrl()` during the round-1 review fix (F2).

### Challenges faced
- Attempt 1 failed: `docker/local/Dockerfile.anchor`'s original `debian:bookworm-slim` base builds natively for linux/arm64 on this Apple Silicon host, and neither Solana CLI nor Anchor CLI publish linux/arm64 binaries; the plan's mega-installer script 404s on the prebuilt binary and falls back to an interactive prompt that panics with no TTY.

## Review fixes

- **F1 (round 1, important):** `on-chain-client`'s vitest suite was never executed by CI/Justfile — wired `pnpm --filter on-chain-client run test` into both, without broadening to a blind `pnpm -r` fan-out (which would have also fired infra-dependent `on-chain-program-e2e`/`e2e` suites standalone and broken them).
- **F2 (round 1, important):** `noop.ts` reinvented a fragile WS-URL derivation (scheme-swap only, silently wrong for the local-validator 8899→8900 convention or any port-specific case) in violation of the project's single-connection-utility rule. Centralized as `deriveWsUrl()` in `connection.ts`, exposed via a new `rpcSubscriptions` field on `SolanaContext`, with TDD coverage for local/public/other-port cases.
- **F3 (round 1, important):** `surfpool/surfpool:latest` was the one unpinned dependency in an otherwise precisely-pinned stack. Pinned by digest in both compose files.
- **F4 (round 1, important):** CircleCI's `cargo-test` job installed the full Anchor/Solana toolchain for a job that never shells out to any of it. Split into a minimal `install_rust` command used by `cargo-test`, keeping the full toolchain install only where `on-chain-program-e2e` genuinely needs it.
- **Verification-failure fix (post-round-1, not a review-round issue):** a live re-verify caught a real bug — the program's deployed identity depended on a gitignored, host-mutable keypair (`target/deploy/*-keypair.json`) that silently regenerates and drifts from the committed `declare_id!` whenever it goes missing, which had in fact happened across the many agent runs sharing this host's checkout. Root-caused and fixed by committing a fixed, throwaway, non-secret local-dev-only deploy keypair (same precedent as the already-committed `SYSTEM_ADMIN_SECRET_KEY`), seeded deterministically into `target/deploy/` by the Docker build before `anchor build` runs. Proven reproducible by deleting the local keypair and rebuilding fresh — same address every time — and independently re-confirmed by a separate verifier agent via a real on-chain transaction.
- **F1 (round 2, important):** A CI comment justifying tolerance of `cimg/base:current`'s moving tag via anchor-cli's glibc requirement was left stale after F4's fix removed that dependency from the job. Rewrote the comment to remove the now-contradicted rationale.
- **F5 (round 3, important):** CI's `typecheck`/`lint` jobs silently no-op'd for 3 of 4 TypeScript packages (`packages/on-chain-client`, `apps/on-chain-program-e2e`, `apps/e2e` — the last with no `tsconfig.json` at all) because only `apps/frontend` had the scripts wired. This let a real, currently-shipping `TS2345` error slip through in `on-chain-program-e2e`'s test (missing the same `assertIsTransactionWithBlockhashLifetime` narrowing Task 7 already applied correctly in the frontend). Wired typecheck/lint into all three remaining packages, added the missing type-guard call, and fixed a real (not suppressed) ESLint `no-undef` at its root cause by adding Node globals to the shared `eslint.config.mjs` for `.mjs`/`.cjs` files.
