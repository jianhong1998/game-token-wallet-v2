# Move `on-chain-client` from `packages/` into `apps/`

## Context

`packages/on-chain-client` (Anchor IDL + Codama-generated TS client) currently
sits under a top-level `packages/` directory, separate from `apps/`. This was
a deliberate decision recorded in
[002-decisions.md](../../technical-related/codebase-structure/002-decisions.md)
Q2 during the original `/grill-me` session (ticket 001): "split later only if
a second genuinely independent consumer appears."

No second consumer has appeared, and `packages/` has never held more than
this one workspace member. Revisiting the decision: is a second top-level
source root pulling its weight for exactly one entry?

## Questions & Answers

**Q: What's driving the move — why should `on-chain-client` live under
`apps/` instead of `packages/`?**
A: Consistency — `apps/` should be the one "all code lives here" root. With
only one package ever under `packages/`, a second top-level source dir just
for it is unjustified structure. It's not a general-purpose library either —
it's generated by `on-chain-program` and consumed only by `frontend` and
`on-chain-program-e2e` inside this repo, so it's project-internal glue, not
a reusable/publishable package.

**Q: Should the `packages/*` workspace glob be dropped entirely, or kept
available for a future genuinely-external package?**
A: Drop it entirely. If a genuinely reusable/publishable package shows up
later, that's a new decision to make then — don't keep dead structure around
for a hypothetical (YAGNI).

## Decision

Move `packages/on-chain-client` → `apps/on-chain-client` via `git mv`
(preserves file history). Remove the `packages/*` line from
`pnpm-workspace.yaml` and delete the (now-empty) `packages/` directory.

### Why this is low-risk

`on-chain-client` is consumed everywhere via the pnpm workspace protocol by
**package name**, not path — `"on-chain-client": "workspace:*"` in
`apps/frontend/package.json` and `apps/on-chain-program-e2e/package.json`,
and `pnpm --filter on-chain-client run ...` in `.circleci/config.yml` and
`docker-compose.e2e.yml`. None of that needs to change.

`packages/on-chain-client/scripts/generate-client.mjs` resolves the Anchor
IDL path via `join(__dirname, "..", "..", "..", "apps", "on-chain-program",
...)`. Both `packages/on-chain-client/scripts` and
`apps/on-chain-client/scripts` are the same depth from repo root (3
segments), so this relative-path arithmetic keeps resolving correctly with
no edit.

No `tsconfig.json` `paths` alias points at `packages/` — only
`apps/frontend/tsconfig.json`'s `@/*` alias exists, and it's self-contained.

### Explicitly out of scope / unaffected

- `.circleci/config.yml` — filters by package name only, no path references.
- `apps/on-chain-program-e2e`, `apps/frontend` — depend via `workspace:*`,
  no path references.
- `.loop-logs/**` — historical execution logs from the original scaffold
  task; left untouched, not a current-state doc.
- `pnpm-lock.yaml` — not hand-edited; regenerated via `pnpm install`.

## Verification

Per this repo's Done Means (docker volume/path changes are exactly the kind
of thing that can silently break at runtime while typecheck stays green):

1. `pnpm install` (lockfile regen), then `pnpm -r lint`, `pnpm -r typecheck`,
   `pnpm -r test`, `pnpm -r build`.
2. Boot the local dev stack (`docker-compose.yml`) and confirm the noop
   transaction flow still works end-to-end (the same check described in
   `README.md`: open the app, send the noop transaction, confirm a real
   signature renders) — this exercises the renamed volume mounts and the
   `program-deploy` → codegen path.
3. `docker compose -f docker-compose.e2e.yml config` (or a full run) to
   confirm the renamed volume + `COPY` path in `docker/deployment/Dockerfile`
   still resolve.
