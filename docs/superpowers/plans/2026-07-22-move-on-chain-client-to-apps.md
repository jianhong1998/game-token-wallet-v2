# Move on-chain-client into apps/ Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move the `on-chain-client` workspace package from `packages/on-chain-client` to `apps/on-chain-client`, drop the now-unused `packages/*` workspace glob entirely, and update every path reference (Docker, docs) so `apps/` is the single root for all workspace code.

**Architecture:** Pure structural move — no source code changes to `on-chain-client` itself. `git mv` the directory, remove the `packages/*` glob from `pnpm-workspace.yaml`, then fix the handful of places that hardcode the `packages/on-chain-client` path (two docker-compose files, one Dockerfile, six docs). Everything that references the package by **name** (`workspace:*` deps, `pnpm --filter on-chain-client`) needs no change.

**Tech Stack:** pnpm workspaces, Docker Compose, Anchor/Codama codegen (unaffected — see spec).

**Spec:** [`docs/superpowers/specs/2026-07-22-move-on-chain-client-to-apps-design.md`](../specs/2026-07-22-move-on-chain-client-to-apps-design.md)

## Global Constraints

- Consume `on-chain-client` only by workspace **name**, never introduce path-based deps — matches existing convention.
- Drop `packages/*` from `pnpm-workspace.yaml` entirely — no dead glob kept "for later" (YAGNI, per spec).
- `docs/technical-related/codebase-structure/002-decisions.md` is a historical decision log — **append** a new entry, never edit Q2 in place.
- `.loop-logs/**` is historical execution output — do not touch.
- All work happens on branch `chore/000-move-on-chain-client-to-apps` (already created off `main`, currently holds only the committed design spec).
- Runtime behavior touches Docker volume mounts and a multi-stage `COPY` path — per this repo's Done Means, must be verified by actually booting the stacks, not just by lint/typecheck passing.

- [ ] **Step 5: Append a new decision entry to 002-decisions.md — do not edit Q2**

Add this new section at the end of the file (after Q13), preserving Q13's content exactly as-is above it:

```markdown

---

## Q14: `packages/on-chain-client` turned out to be the only thing ever under `packages/` — does that top-level split still earn its keep?

**Answer:** No — moved to `apps/on-chain-client`; `packages/` is removed.

**Decision:** `on-chain-client` moves from `packages/on-chain-client` to
`apps/on-chain-client` (see
[2026-07-22-move-on-chain-client-to-apps-design.md](../../superpowers/specs/2026-07-22-move-on-chain-client-to-apps-design.md)).
The `packages/*` line is dropped from `pnpm-workspace.yaml` entirely rather
than kept empty for a hypothetical future package.

**Reason:** Q2 reserved `packages/` for shared code, with an explicit
"split later only if a second genuinely independent consumer appears"
condition. No second consumer ever appeared, and `on-chain-client` is not a
general-purpose library anyway — it's generated by `on-chain-program` and
consumed only by `frontend` and `on-chain-program-e2e` inside this repo.
A second top-level source root for exactly one project-internal package
wasn't earning its keep. `apps/` becomes the single root for all workspace
code; if a genuinely reusable/publishable package appears later, that's a
new decision to make then, not a reason to keep the glob now (YAGNI).
```
